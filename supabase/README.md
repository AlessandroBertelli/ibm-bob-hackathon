# Supabase

Database schema, RLS, RPCs, Realtime publication, Storage bucket, and pg_cron jobs for atavola — all in one consolidated migration.

## Apply the schema

The fastest path is to copy [migrations/0001_init.sql](migrations/0001_init.sql) into the **SQL Editor** in your Supabase project dashboard and run it.

If you have the Supabase CLI installed and your project linked, you can also run:

```bash
supabase db push
```

The file is idempotent (`create … if not exists`, `create or replace`, `drop policy if exists` before each `create policy`) — re-running it is safe.

## What the migration creates

### Tables

| Table | Purpose |
|---|---|
| `profiles` | Auto-created on auth signup via the `handle_new_user` trigger. |
| `saved_meals` | "My Food". `user_id` is `ON DELETE SET NULL` so deleted accounts leave anonymised recipes behind. |
| `sessions` | One per host run. `share_token` is the public link. `expires_at` is 30 days out. |
| `session_meals` | The 4 generated cards per session. |
| `guests` | Anonymous voters. `guest_token` proves identity for `cast_vote`. |
| `votes` | Yes / no per (guest, meal). |
| `rate_log` | Sliding-window counters keyed by `(actor, scope)` for `check_rate`. |
| `service_status` | Last outcome per external service (Supabase / OpenRouter / each image provider / Resend). Powers `/api/system/status`. |
| `events` | Lightweight tracking — visits, logins, sessions, votes, saves. Trimmed to last 30 days. |
| `aggregated_stats` | Lifetime counters that survive `cleanup_after_digest`. |
| `error_log` | Recent backend errors; flushed each Monday after the digest emails. |

### RPCs (all SECURITY DEFINER)

| RPC | Used by |
|---|---|
| `cast_vote(guest_token, meal_id, value)` | `POST /api/votes`. Rejects expired or non-`voting` sessions. |
| `ensure_guest(session_id, user_id)` | `POST /api/votes/guest`. Same session-status guard. |
| `check_rate(actor, scope, limit, window)` | `api/_lib/ratelimit.ts` — Postgres-backed sliding-window limiter. |
| `record_service_outcome(service, ok, detail)` | Backend services log every external call. |
| `record_event(kind, payload)` | Tracking endpoints + key code paths. |
| `record_error(area, message, context)` | Catch-blocks in serverless handlers. |
| `get_weekly_digest_data()` | The weekly cron pulls aggregated counters + recent errors. |
| `cleanup_after_digest()` | Trims `events` past 30 days, clears `error_log`. |
| `list_my_sessions(p_user_id)` | `GET /api/sessions/mine` — host history with voter counts. |

### Realtime

`session_meals` and `votes` are added to the `supabase_realtime` publication so the live-results screen subscribes directly via the anon key (RLS gates it).

### Storage

Public-read `meal-images` bucket; only the service role can write. Generated bytes from any image provider land here so the URL we hand back never depends on the provider's cache.

### pg_cron

`atavola-cleanup-expired-sessions` runs daily at **03:15 UTC** (`15 3 * * *`). It deletes sessions where `expires_at < now()`. Cascades drop guests, votes, and session_meals with them. Saved meals survive because `saved_meals.source_session_meal_id` is `ON DELETE SET NULL`.

The migration enables the `pg_cron` extension first; if your project doesn't have it, the migration logs a notice and the rest still applies — turn on `pg_cron` under **Database → Extensions** and re-run.

## RLS in one sentence

`sessions`, `session_meals`, `guests`, `votes` are publicly readable so the share link works without auth; everything else is owner-only or service-role-only. See the `create policy` blocks in the migration for the exact rules.

## Auth & email

Magic-link emails are sent by Supabase Auth. To use Resend as the SMTP provider, configure it under **Authentication → Email → SMTP Settings** in the dashboard. The default English templates work — keep `{{ .ConfirmationURL }}` as the link placeholder.

The weekly digest is sent separately via the Resend HTTP API from the Vercel cron `/api/_cron/weekly-stats`. That uses `RESEND_API_KEY` + `EMAIL_FROM` directly and does not go through Supabase.

## After running the migration

1. Set the **Site URL** under **Authentication → URL Configuration** to your frontend origin (e.g. `https://atavola.vercel.app`).
2. Add the auth verify path to **Redirect URLs**: `${SITE_URL}/auth/verify`.
3. Confirm the `meal-images` bucket exists under **Storage**.
4. Confirm `atavola-cleanup-expired-sessions` is listed under **Database → Cron Jobs**.
